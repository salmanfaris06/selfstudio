## 1. Epic Review

Epic 7 menyelesaikan managed gPhoto2 camera tethering untuk selfstudio. Scope utama adalah membuat kamera fisik dapat ditemukan, di-assign ke station, dijalankan sebagai tether listener yang supervised, divalidasi readiness/test capture, ditampilkan dan dikontrol dari dashboard, dipulihkan saat disconnect/restart, dan diverifikasi lewat guided hardware smoke test artifact.

Prinsip arsitektur yang dipertahankan sepanjang epic: **gPhoto2/tether hanya menulis JPG ke `station.input_folder`; folder watcher/ingestion existing tetap menjadi source of truth** untuk routing session, save original, LUT processing, quarantine, dan Google Drive upload. Tidak ada shortcut dari camera event langsung ke photo/session/upload record.

### Story Selesai

- 7.1 Detect and Assign gPhoto2 Cameras to Stations — done
- 7.2 Supervise gPhoto2 Tether Listener per Station — done
- 7.3 Add Camera Readiness Checks and Test Capture Validation — done
- 7.4 Show Live Camera/Tether Status and Controls in the Dashboard — done
- 7.5 Recover and Reconnect gPhoto2 Tethering During Event — done
- 7.6 Run Real Hardware gPhoto2 Event Smoke Test — done

### Yang Dibangun

1. **Discovery dan assignment kamera gPhoto2**
   - Backend Go menambahkan discovery gPhoto2 native/WSL dengan safe diagnostics.
   - Assignment kamera persisted di station config dan reusable setelah restart.
   - Duplicate camera assignment ditolak di backend.
   - Safe action tersedia untuk kondisi seperti `INSTALL_GPHOTO2`, `CHECK_WSL`, `CHECK_USBIPD`, `CONNECT_CAMERA`, dan `CHECK_CAMERA_USB_MODE`.

2. **Supervised tether listener per station**
   - Start/stop/status listener per station via authenticated trusted-origin API.
   - Satu logical listener per station dengan duplicate-start guard.
   - Stop hanya menghentikan process milik station terkait, bukan global gPhoto2 process.
   - File output confined ke input folder station dan ingestion tidak dipanggil langsung.

3. **Camera readiness dan test capture validation**
   - Readiness station diperluas dengan camera assignment, gPhoto2 availability, camera connected, tether listener, writable input folder, dan last test capture.
   - Camera readiness dapat menjadi optional/required untuk session start.
   - Test capture explicit operator action, expected-file-only, JPEG/stable validation, dan tidak bergantung pada Drive credentials.

4. **Dashboard live camera/tether controls**
   - Station card menampilkan assigned/unassigned, connected/disconnected, listener running/stopped/error, last capture, last downloaded filename, dan next action.
   - Dashboard memiliki Start/Stop/Retry listener dan Run Test Capture.
   - SSE invalidation ditambahkan untuk camera/tether updates.
   - UI memisahkan camera/tether status dari local processing dan Google Drive upload.

5. **Recovery dan reconnect hardening**
   - Desired listener state dan auto-restart setting persisted secara lokal.
   - Recovery coordinator dengan backoff mencegah tight restart loop.
   - Startup recovery hanya mencoba restart jika desired running dan auto-restart enabled.
   - Manual retry tetap safe, authenticated, trusted-origin, dan station-scoped.

6. **Guided hardware smoke test artifact**
   - Endpoint/report untuk hardware smoke local-only/Drive optional/Drive verify.
   - Report berisi safe fields: station, camera summary, basename file, step status, failure code/action, dan next action.
   - Report disimpan sebagai local diagnostic artifact tanpa overwrite/destructive cleanup.
   - No-delete/no-privileged-action guardrails didokumentasikan dan diuji.

## Yang Berjalan Baik

1. **Boundary arsitektur tetap konsisten**
   - Browser tidak menjalankan command, tidak membaca filesystem lokal, dan tidak menerima raw diagnostic.
   - Go agent tetap menjadi pemilik process gPhoto2, station config, readiness, recovery, report, SSE, dan activity.

2. **Source of truth ingestion tidak berubah**
   - Listener, test capture, recovery, dan hardware smoke semuanya menjaga prinsip: file masuk ke input folder, lalu watcher existing yang memproses.
   - Ini mengurangi risiko duplicate processing, session boundary salah, atau upload langsung sebelum local completion.

3. **Safety-first command execution**
   - Command dibangun dari allowlist, bukan shell string.
   - Forbidden actions seperti install package, `usbipd bind/attach`, driver/Zadig changes, global taskkill/pkill, dan delete artifacts tidak dijalankan otomatis.

4. **Review feedback menghasilkan hardening signifikan**
   - Story 7.1 memperbaiki usbipd diagnostic path, parser non-USB/PTP, dan API test coverage.
   - Story 7.2 memperbaiki start/stop race dan validasi station scope.
   - Story 7.3 memperbaiki timeout test capture, readiness aggregate, raw UI exposure, safe error details, API tests, dan filename collision.
   - Story 7.4 memperkuat redaction dashboard untuk listener message dan command/path-like diagnostics.
   - Story 7.5 memperkuat recovery setting recheck, one-shot manual retry saat auto-restart disabled, dan recovery validation.
   - Story 7.6 memperbaiki continue mode, active session safety, Drive verify semantics, report persistence, atomic writer, stable JPG validation, dan nil assignment safety.

5. **Operator UX menjadi lebih actionable**
   - Dashboard menampilkan next action berbasis safe action code.
   - Camera/tether status dipisahkan dari local processing dan Drive upload, mengurangi kebingungan saat event pressure.

## Tantangan dan Pelajaran

### 1. Hardware integration tidak boleh dianggap sama dengan unit/API success

Automated tests bisa membuktikan parser, runner, API contract, recovery, no-leak, no-delete, dan UI safety. Namun gPhoto2 + WSL + usbipd + kamera fisik tetap punya variabel real-world: mode kamera, port berubah, kamera busy, device attach, kabel, dan OS permission.

**Lesson:** Automated validation adalah necessary, bukan sufficient. Real hardware smoke harus menjadi gate operasional sebelum event.

### 2. Raw diagnostics mudah bocor jika internal model langsung ditampilkan

Beberapa review menemukan potensi raw port, identity key, source path, command-like diagnostic, dan full test object di error details.

**Lesson:** API/UI perlu safe DTO dan allowlisted display fields. Jangan expose persistence/runtime object langsung ke operator UI.

### 3. Process lifecycle butuh race-aware design

Start/stop listener, unexpected exit, startup recovery, manual retry, duplicate click, dan auto-restart bisa saling bertabrakan.

**Lesson:** Side-effect worker harus singleton/shared, station-scoped, mutex/guard jelas, desired state persisted, dan recovery memakai backoff.

### 4. Recovery harus memodelkan uncertainty secara jujur

Jika listener exit, kamera disconnect, atau app restart, sistem tidak boleh berpura-pura ready. Recovery boleh mencoba hal yang aman, tetapi operator tetap harus mendapat attention state dan next action.

**Lesson:** Attention state dengan safe action lebih baik daripada success palsu.

### 5. Hardware smoke harus aman terhadap session customer aktif

Smoke test dapat menghasilkan foto nyata. Jika active session customer sedang berjalan, smoke bisa mencemari session kecuali operator secara eksplisit mengizinkan atau dedicated smoke session dibuat.

**Lesson:** Diagnostic tooling di production harus punya guardrail terhadap customer data/workflow, bukan hanya teknis berhasil.

## Risiko Tersisa

1. **Port/identity kamera dapat berubah setelah unplug/replug**
   - Assignment berbasis runtime/port/name cukup untuk MVP, tetapi serial-level stability belum dijamin tanpa capability probe yang aman.
   - Follow-up: evaluasi safe serial extraction bila kamera mendukung dan tidak membocorkan data sensitif.

2. **WSL/usbipd setup tetap manual dan environment-sensitive**
   - Sistem sengaja tidak menjalankan privileged bind/attach/install.
   - Follow-up: operator runbook harus jelas untuk WSL, usbipd, camera mode, dan recovery action.

3. **Real event behavior belum dibuktikan dengan kamera fisik**
   - Automated tests tidak menggantikan kabel/kamera/OS/driver/gPhoto2 real path.
   - Follow-up: jalankan hardware smoke local-only dan Drive verify bila Drive sudah configured.

4. **Dashboard kompleksitas meningkat**
   - Operator sekarang melihat session, watcher, processing, quarantine, Drive, camera/tether, recovery, dan smoke status.
   - Follow-up: lakukan walkthrough operator agar action labels dipahami.

5. **Recovery auto-restart perlu kebijakan operasional**
   - Default safe adalah disabled; enabling auto-restart harus diputuskan per event/station.
   - Follow-up: tentukan kapan auto-restart dinyalakan dan bagaimana operator memonitor paused/backoff state.
