   - Pending/retryable jobs resume setelah durable normalization.

## Tantangan dan Pelajaran

### 1. Upload retry bukan sekadar menjalankan ulang upload

Retry cloud perlu menjaga job identity, object key, remote generation/identity, precondition conflict, local file availability, dan guard concurrency. Tanpa itu, retry dapat membuat duplicate object atau overwrite yang tidak aman.

**Lesson:** Retry remote storage harus didesain sebagai state machine idempotent, bukan loop sederhana.

### 2. Worker yang tercopy by value bisa merusak duplicate guard

Review Story 6.4 menemukan worker/guard sempat tersalin sehingga scheduler dan API/manual retry dapat punya guard berbeda. Ini sistemik: value copy terhadap struct berisi mutex/map berbahaya untuk concurrency control.

**Lesson:** Worker/guard state yang mengontrol side effect harus shared lewat pointer atau dependency tunggal yang tidak mudah tercopy, dan harus ada concurrent test.

### 3. Safe DTO perlu dipikirkan terpisah dari persistence model

Persisted `FileUploadJob` membutuhkan `local_path`, tetapi browser tidak perlu full path tersebut. Mengembalikan persistence model langsung ke API membuat kebocoran path customer.

**Lesson:** Jangan expose internal persistence model langsung. Buat response DTO aman untuk API/SSE/UI.

### 4. Success event harus menunggu durable save

Beberapa follow-up menegaskan pola penting: jangan publish `authorized`, `target_resolved`, `file_uploaded`, atau `recovered` sebelum state berhasil disimpan. Kalau save gagal, event success akan membuat dashboard dan operator percaya pada state yang tidak recoverable.

**Lesson:** Durable state dulu, baru event/activity success.

### 5. Recovery harus jujur terhadap state tidak pasti

Stale `uploading/retrying` setelah restart tidak boleh diasumsikan uploaded. Uploaded jobs tanpa verifier juga tidak boleh disebut verified. Story 6.5 memperbaiki ini dengan `unverified_uploaded`/remote verifier wiring dan safe action saat object perlu dicek.

**Lesson:** Saat restart, uncertainty adalah state nyata yang harus dimodelkan, bukan disembunyikan.

## Review Feedback Patterns

Tema feedback berulang sepanjang Epic 6:

- **Credential safety:** secret tidak boleh muncul di response, event, activity, OpenAPI example, atau frontend state.
- **Persistence-before-event:** jangan klaim success bila save/load runtime state gagal.
- **Idempotency/de-dup:** session target, file job, retry, and recovery harus memakai identity deterministic.
- **Concurrency guard:** auto retry, manual retry, scheduler, recovery, dan double-click UI harus share guard yang sama.
- **UI reachability:** locked/completed sessions harus tetap terlihat untuk cloud action/status.
- **Safe API boundary:** expose `local_file_name`, bukan full `local_path`; expose safe cloud error/action, bukan raw privileged error.
- **Truthful aggregate status:** partial failed/pending/retrying/uploaded harus terlihat jujur, tidak diringkas menjadi success palsu.
