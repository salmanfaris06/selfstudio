## 1. Epic Review

### Ringkasan Epic

Epic 1 membangun fondasi lokal selfstudio agar operator dapat menjalankan aplikasi di Windows, mengakses dashboard melalui local PIN gate, memakai kontrak API/SSE yang konsisten, melihat health dashboard shell, dan melihat activity log awal.

**Stories selesai:** 5/5

1. Story 1.1 — Create Windows Local App Skeleton
2. Story 1.2 — Add Local PIN Gate
3. Story 1.3 — Establish API, Error, and SSE Contract Foundation
4. Story 1.4 — Create Health Dashboard Shell
5. Story 1.5 — Record and View Operator Activity Log Foundation

### Hasil Utama

- Monorepo final mulai terbentuk di `apps/web`, `apps/agent`, `docs/api`, `supabase/migrations`, `scripts`, dan `local-data`.
- Go agent menjadi backend lokal utama untuk API, auth, SSE, health, dan activity foundation.
- Next.js dashboard memiliki auth gate, TanStack Query provider, health shell, SSE integration, dan activity log UI.
- API contract memakai pola `{data}` dan `{error:{code,message,action,details}}` secara konsisten.
- SSE foundation sudah protected oleh local session dan memakai event envelope standar.
- Local PIN gate sudah lebih aman dari versi awal: session TTL, throttling, credentialed CORS, validasi JSON, placeholder PIN rejection, dan session-cookie safeguards.
- Activity log foundation sudah protected, bounded, punya filter action type, dan menghindari secret leakage.

### Kualitas dan Validasi

Validasi berulang yang terbukti:

- `cd apps/agent && go test ./...` — passed setelah patch tiap story.
- `cd apps/web && npm run typecheck && npm run build` — passed setelah patch tiap story.
- Manual smoke dilakukan untuk health, auth, SSE, dashboard, CORS, dan activity endpoints.

Catatan tooling:

- `biome_check` gagal dari harness dengan `spawn EINVAL`; ini dicatat sebagai masalah harness/tooling, bukan kegagalan kode.
- `npm install @tanstack/react-query` melaporkan vulnerabilities; `npm audit fix --force` sengaja tidak dijalankan karena berpotensi breaking.
- Satu kali Go test terkena transient Windows `Access is denied`, lalu lolos setelah retry.

## 2. Pola Temuan dari Review

### Tema Berulang

1. **Keamanan boundary lokal perlu eksplisit.**
   - CORS, cookie auth, CSRF/Origin guard, forwarded headers, session expiry, dan brute-force throttling muncul sebagai review findings.
   - Lesson: walaupun MVP lokal, endpoint state-changing tetap harus diperlakukan sebagai attack surface.

2. **Public endpoint tidak boleh punya side effect audit penting.**
   - Story 1.5 menunjukkan `/api/health` public bisa menjadi sumber spam activity jika langsung dicatat.
   - Lesson: telemetry, health, audit, dan operator action perlu dipisahkan jelas.

3. **Kontrak API harus divalidasi dua arah.**
   - Backend perlu OpenAPI jelas; frontend perlu hardening terhadap malformed success/error payload.
   - Lesson: typed helper saja tidak cukup; runtime validation minimal tetap penting untuk UI operator.

4. **SSE punya edge case khusus.**
   - `WriteTimeout`, keepalive write failures, event ID format, event type validation, dan auth expiry behavior semua perlu diperhatikan.
   - Lesson: realtime foundation harus diuji sebagai long-lived connection, bukan sekadar HTTP biasa.

5. **Windows-first bukan detail kecil.**
   - Next.js config compatibility, PowerShell quoting, local path resolution, transient test binary access, dan scripts env propagation semua muncul.
   - Lesson: setiap story yang menyentuh runtime lokal harus diuji dengan asumsi Windows sebagai target utama.

### Perbaikan Proses yang Berhasil

- Code review adversarial paralel sangat efektif menangkap issue yang berbeda jenis:
  - Blind Hunter menangkap security/API basics dari diff.
  - Edge Case Hunter menangkap boundary cases dan runtime UI crash.
  - Acceptance Auditor menangkap mismatch terhadap AC.
- Pola “implement → validate → adversarial review → patch → validate ulang” meningkatkan kualitas setiap story.
- Story file sebagai sumber konteks bekerja baik karena findings, file list, debug log, dan change log terdokumentasi dekat dengan scope story.

## 3. Technical Debt dan Risiko Tersisa

### Debt yang Disengaja / Belum Masuk Scope

- Activity log masih in-memory; persistence Supabase belum dibuat.
- Health database/worker/disk masih placeholder; real probes masuk Epic 2.
- Root TypeScript camera-input spike masih dipertahankan sebagai prototype dan belum direkonsiliasi dengan final Go worker.
- No full Playwright/E2E test untuk dashboard auth/health/activity flow.
- No generated TypeScript client dari OpenAPI; kontrak masih dijaga manual melalui types dan tests.
- `project-context.md` belum ada; beberapa workflow mencarinya tetapi tidak menemukan file tersebut.

### Risiko untuk Epic 2

- Station config akan memperluas data model; perlu mulai memutuskan persistence boundary lebih tegas.
- Readiness checks akan mengubah health placeholder menjadi checks nyata; perlu hindari mencampur public health dengan operator-triggered checks.
- Activity log perlu station/session references nyata dan filter yang lebih berguna.
- Path validation dan Windows filesystem behavior akan menjadi pusat risiko.
- Jika Supabase schema belum diperkenalkan di awal Epic 2, config bisa terjebak terlalu lama di in-memory/file-only foundation.

## 4. Apa yang Berjalan Baik

- Arsitektur Go + Next.js terbukti cocok untuk pembagian tanggung jawab: Go untuk local service/worker, Next.js untuk operator UI.
- Local PIN gate bisa dibuat cukup aman tanpa membawa kompleksitas Supabase Auth di MVP awal.
- OpenAPI sebagai kontrak awal membantu menjaga endpoint konsisten.
- TanStack Query + SSE integration memberi pola yang siap dipakai fitur station/session berikutnya.
- Local shadcn/ui-style primitives cukup untuk memulai tanpa mengunci project pada registry setup terlalu dini.
- Manual smoke testing setelah patch review menangkap behavior yang tidak selalu tertutup unit test.

## 5. Apa yang Perlu Diperbaiki

- Tambahkan project context file agar agent berikutnya tidak kehilangan keputusan arsitektur penting.
- Hindari endpoint public dengan side effect state/audit kecuali eksplisit dan rate-limited.
- Tambahkan test route-level untuk setiap protected endpoint, bukan hanya handler-level.
- Pertimbangkan helper standar untuk state-changing route guard agar Origin/CSRF behavior tidak perlu diingat manual.
- Mulai pisahkan action audit, health telemetry, readiness checks, dan operator commands secara eksplisit.
- Kurangi contract drift dengan OpenAPI linting atau client generation ketika API mulai membesar.

4. Story 2.4 — Refresh and Reconnect Station Health

6. Story 2.6 — Run Event Readiness Checklist

### Dependencies dari Epic 1

- Auth gate dan protected API sudah tersedia untuk station config endpoints.
- API/error response helpers sudah siap dipakai.
- SSE foundation siap untuk station/readiness update events.
- Health dashboard shell siap diperluas dari placeholder ke station/readiness indicators.
- Activity log foundation siap mencatat config changes dan readiness/reconnect actions.

### Persiapan Penting Sebelum/di Awal Story 2.1

- Putuskan persistence untuk station configuration.
- Tambahkan domain package Go untuk station config dan validation rules.
- Pastikan route guard pattern konsisten untuk all protected config mutations.
- Desain `station_id`, station status vocabulary, dan action types sebelum UI terlalu jauh.
- Tetap Windows-first untuk folder picker/path validation assumptions.
