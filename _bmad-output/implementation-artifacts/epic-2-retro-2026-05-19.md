## 1. Epic Review

### Ringkasan Hasil

Epic 2 menyelesaikan fondasi setup dan kesiapan tiga camera station untuk operasi event. Operator sekarang bisa mengonfigurasi station, memvalidasi folder/LUT/device, menjalankan checklist readiness event, backup/restore konfigurasi, menjalankan watch validation, dan refresh station health tanpa restart penuh.

### Story yang Selesai

- 2.1 Configure Three Camera Stations
- 2.2 Validate Station Paths, LUTs, and Device Readiness
- 2.3 Run Event Readiness Checklist — diimplementasikan sebagai `2-3-create-event-readiness-checklist`
- 2.4 Backup and Restore Station Configuration — diimplementasikan sebagai `2-4-save-and-restore-station-configuration`
- 2.5 Run Test Capture Watch Validation — diimplementasikan sebagai `2-5-run-test-capture-watch-validation`
- 2.6 Refresh and Reconnect Station Health — diimplementasikan sebagai `2-6-refresh-and-reconnect-station-health`

Catatan: Sprint status sempat mengalami numbering drift antara backlog resmi dan file implementasi aktual. Drift sudah disinkronkan dengan alias komentar di `sprint-status.yaml` agar Epic 2 bisa ditutup tanpa duplikasi story.

### Yang Berjalan Baik

- Pola backend Go untuk domain station/readiness makin konsisten: store, validator, handler, tests.
- Mutation endpoint makin aman karena selalu membawa local auth dan trusted Origin guard.
- Frontend memakai TanStack Query + SSE invalidation sebagai pola standar.
- Runtime validation frontend mencegah UI menerima response backend yang salah bentuk.
- OpenAPI diperbarui setiap kali contract berubah.
- Review adversarial menangkap beberapa risiko penting sebelum story ditutup.

### Yang Perlu Diperbaiki

- Numbering drift membuat judge loop berhenti karena backlog terlihat belum selesai meskipun fitur sudah diimplementasikan.
- Beberapa validator frontend terlalu mengunci placeholder Epic 2 dan perlu dilonggarkan saat Epic 3 mulai membuka session start nyata.
- Ada deferred security/UX contract terkait auth expiry, login/logout origin/auth, dan health endpoint publik.
- Device readiness masih `unknown`; ini benar untuk MVP, tetapi operator perlu konteks yang jelas saat masuk Epic 3.

### Risiko yang Dibawa ke Epic 3

- Session start harus menghormati readiness result dan tidak boleh menganggap placeholder/unknown sebagai ready.
- Event readiness saat ini masih memblokir start session karena session start belum tersedia; Epic 3 harus mengubah contract ini dengan hati-hati.
- Activity log masih in-memory; restart akan menghapus audit trail sampai persistence dikerjakan.
- File watcher produksi belum ada; watch validation hanya validation-only polling.
- Frontend auth-expiry handling belum penuh; SSE error bisa membuat operator perlu login ulang manual.

## 2. Next Epic Preparation — Epic 3

Epic 3 akan membuka Live Session Control Room. Fokus utama: operator bisa membuat satu session aktif per station, melihat live station cards, mengakhiri session, dan mempertahankan state setelah restart.

### Rekomendasi Teknis untuk Epic 3

1. Buat domain `sessions` di Go sebelum UI besar dibuat.
2. Simpan session state secara durable sejak story pertama Epic 3, minimal local JSON atomic seperti station config.
3. Enforce one-active-session-per-station di backend, bukan hanya frontend.
4. Snapshot station config saat session start: background, LUT, output rule, station name.
5. SSE event awal yang disarankan:
   - `session.started`
   - `session.start_failed`
   - `session.ended`
   - `session.timer_updated`
   - `station.live_status_updated`
6. Jangan membuka photo routing dulu; session start boleh menciptakan container state tanpa ingestion.
7. Update event readiness agar `session_start_available` bisa `true` hanya jika required station checks aman dan session service siap.

### Action Items

- [ ] Epic 3 Story 3.1 harus membawa pelajaran trusted Origin, runtime validation, OpenAPI, activity, dan SSE invalidation.
- [ ] Tambahkan deferred item untuk melonggarkan validator readiness ketika session start sudah tersedia.
- [ ] Pastikan sprint status untuk Epic 3 tidak mengulang numbering drift: gunakan key resmi dari epics.md.
- [ ] Pertimbangkan persistence activity log sebelum fitur session menjadi sensitif secara operasional.
