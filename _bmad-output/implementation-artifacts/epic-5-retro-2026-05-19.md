### Ringkasan Outcome

Epic 5 memenuhi tujuan utama FR31–FR39, FR43, FR45–FR48, dan FR54 dalam scope processing/local delivery:

- Original JPG disimpan ke folder deterministic customer/order/station sebelum graded/LUT processing dimulai.
- `local_original_path` dan status original-save dipersist dan direkonsiliasi setelah restart.
- Graded output dibuat dari `local_original_path`, bukan dari `source_path`, sehingga original-first invariant terjaga.
- Missing/invalid LUT, processor unavailable, dan processing failure menjadi failed/retryable tanpa menghapus original.
- Queue/status processing tersedia via API/UI dengan status bucket, current job, failure reason, attempt count, dan last update.
- Retry otomatis dibatasi sampai tiga attempt total; manual retry tersedia sebagai operator action.
- Recovery startup memverifikasi original/graded files, menghormati retry limit, memakai shared processing guard, dan tidak menjalankan long processing sebelum dashboard/API tersedia.
- SSE/activity events memakai safe IDs seperti `photo_id`, bukan raw path sebagai entity id.

## Yang Berjalan Baik

1. **Original-first invariant berhasil dijadikan acceptance gate**
   - Story 5.1 memastikan original disimpan dan diverifikasi sebelum graded job eligible.
   - Story 5.2 menegaskan graded processing memakai `local_original_path`, bukan `source_path`.
   - Recovery Story 5.5 memverifikasi original lagi sebelum job dilanjutkan.

2. **Collision safety dan idempotency makin kuat**
   - Filename deterministic memakai `photo_id` sehingga restart/retry tetap menuju path yang sama.
   - Existing original dengan size sama tidak langsung diterima; byte/content comparison ditambahkan.
   - Existing graded target yang tidak terverifikasi ditolak, bukan dianggap processed.

3. **Processing menjadi observable untuk operator**
   - Queue endpoint dan UI menampilkan bucket status, current/recent jobs, attempt count, failure reason, dan manual retry CTA.
   - SSE invalidation memperbarui queue/status saat processing start/success/failure/recovery.

4. **Retry/recovery memakai jalur bersama**
   - Ingestion, quarantine assignment, manual retry, dan recovery memakai shared runner/guard.
   - Ini mengurangi risiko race condition dan duplicate attempt untuk photo yang sama.

5. **Review feedback menghasilkan hardening nyata**
   - Beberapa risiko tinggi ditemukan dan ditutup: stale API response setelah save, silent accept existing file, blocking ImageMagick di request handler, summary count dipengaruhi limit, guard otomatis/manual terpisah, serta startup recovery pre-listen.

## Tantangan dan Pelajaran

### 1. File success harus dibuktikan, bukan diasumsikan

Beberapa review menemukan pola berbahaya: file existing dengan ukuran sama atau path deterministic existing bisa diterima sebagai success tanpa cukup bukti. Itu berisiko membuat metadata menunjuk file yang salah.

**Lesson:** Untuk workflow filesystem, persisted success harus selalu didukung oleh verifikasi yang cukup: exists, readable, non-directory, size/content sesuai, dan path berada di boundary yang aman.

### 2. Request handler bukan tempat untuk image processing berat

Story 5.2 awalnya masih menjalankan graded processing synchronous di ingestion/quarantine flow. Ini dapat membuat dashboard/API terasa blocked saat processor lambat atau ImageMagick timeout.

**Lesson:** Semua processing yang bisa lambat harus masuk background runner/queue dengan event feedback. HTTP mutation boleh memulai/enqueue work, tetapi tidak boleh menunggu image processing selesai.

### 3. Guard concurrency harus global untuk satu photo/job

Story 5.4 menemukan automatic processing dan manual retry sempat memakai guard berbeda. Ini membuka race: manual retry dapat berjalan bersamaan dengan automatic attempt.

**Lesson:** Per-photo processing guard harus shared across semua entry point: ingestion, assignment, manual retry, dan recovery.

### 4. Queue summary harus mencerminkan semua matching jobs

Story 5.3 review menemukan `limit` sempat memengaruhi summary buckets. Itu bisa membuat operator under-estimate backlog.

**Lesson:** Read model untuk operator harus menghitung summary dari seluruh matching set; pagination/limit hanya membatasi item list yang dikembalikan.

### 5. Startup recovery perlu lifecycle awareness

Story 5.5 review menemukan recovery/enqueue dapat terjadi sebelum server/SSE siap, sehingga event penting bisa missed dan dashboard belum available.

**Lesson:** Startup recovery harus memisahkan fast metadata verification dari long-running work, lalu menjadwalkan processing setelah service path siap. Queue API harus dapat menceritakan state terbaru bahkan bila event sebelumnya terlewat.

## Review Feedback Patterns

Tema feedback berulang sepanjang Epic 5:

- **Stale response risk:** API response harus memakai state terbaru setelah side-effect selesai atau queued.
- **Silent success risk:** existing file/path tidak boleh dianggap valid tanpa verification.
- **Blocking risk:** long-running processing tidak boleh berada di HTTP handler atau pre-listen startup path.
- **Concurrency risk:** automatic/manual/recovery paths harus berbagi guard dan policy.
- **Contract accuracy:** OpenAPI, runtime validators, SSE events, dan UI invalidation harus ikut berubah bersama backend behavior.
- **Operator truthfulness:** queue/status harus akurat, actionable, dan tidak under-report karena limit atau missed event.

## Risks Dibawa ke Epic 6

1. **Local completion perlu definisi session-level yang eksplisit**
   - Epic 6 upload harus tahu kapan session local output dianggap complete.
   - Per-photo `processed/failed` sudah ada, tetapi cloud upload perlu aturan: upload hanya processed assets, atau juga original untuk failed graded jobs?

2. **Cloud upload identity harus mengikuti kualitas idempotency Epic 5**
   - Upload retry tidak boleh memakai timestamp volatile.
   - Target object key harus deterministic dan collision-safe seperti local paths.

3. **Credential failure tidak boleh mengganggu capture/local processing**
   - Epic 6 harus menjaga separation: GCS/Drive failure tidak memblokir ingestion, original save, LUT processing, session controls, atau retry lokal.

4. **Queue UX akan makin kompleks**
   - Operator sudah punya processing queue; Epic 6 menambah upload queue.
   - UI harus jelas memisahkan local save/processing status dari cloud upload status agar tidak membingungkan operator.

5. **Recovery scope bertambah**
   - Epic 5 recovery hanya photo processing.
   - Epic 6 harus menambahkan upload job recovery tanpa melemahkan existing processing recovery.

6. **Operational dependency ImageMagick perlu readiness check**
   - Processing failure sudah actionable, tetapi sebelum event sebaiknya readiness checklist mendeteksi processor dependency dan LUT validity lebih awal.

## Significant Discovery Alert

Tidak ada discovery yang membatalkan arah Epic 6. Namun ada guardrail penting yang harus masuk planning Epic 6:

- Cloud upload harus mengikuti prinsip Epic 5: deterministic identity, verification before success, retry-safe, and recoverable.

- Recovery/enqueue upload jobs harus lifecycle-safe: API/dashboard tersedia, queue state truthful, dan events safe.
- Local files must remain source of safety; cloud failure must never mutate/delete local outputs.
