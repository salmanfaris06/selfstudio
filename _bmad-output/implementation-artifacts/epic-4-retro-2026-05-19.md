### Ringkasan Outcome

Epic 4 memenuhi tujuan utama FR21–FR30, FR43, FR44, FR48, dan FR54 dalam scope ingestion/quarantine:

- Sistem mendeteksi `.jpg`/`.jpeg` secara case-insensitive dan menunggu file stabil.

## Yang Berjalan Baik

1. **Server-owned routing berhasil dijaga konsisten**
   - Frontend tidak menentukan session eligibility.
   - Routing, quarantine, assignment, dan recovery tetap berada di Go service.

2. **Duplicate safety makin matang sepanjang epic**
   - Story 4.2 menemukan bahwa identity berbasis `detected_at` tidak stabil.
   - Setelah diperbaiki, identity konsisten memakai `station_id + normalized source_path + source_size_bytes`.
   - Pola yang sama dipakai untuk routed photo dan quarantine.

3. **Operator visibility meningkat bertahap**
   - Live station cards menampilkan routed photo count, quarantine count, dan latest reason.
   - Quarantine review memberi metadata aman, eligible sessions, confirmation, dan feedback aksi.

4. **Review feedback cepat menghasilkan hardening nyata**
   - OpenAPI gaps, stale query invalidation, locked-session eligibility, runtime persistence, dan transactional save semuanya ditangani sebagai review follow-up.

5. **Recovery story menutup risiko besar dari epic sebelumnya**
   - Photo/quarantine store tidak lagi reset kosong saat startup.
   - Startup recovery dan runtime persistence memperkuat NFR9/NFR16.

## Tantangan dan Pelajaran

### 1. Identity idempotency harus dirancang sebelum fitur meluas

Masalah paling penting muncul saat duplicate routed photo bisa terjadi karena identity menyertakan timestamp deteksi. Ini menunjukkan bahwa field yang berubah antar scan/restart tidak boleh menjadi bagian dari identity deduplication.

**Lesson:** Untuk filesystem ingestion, identity harus berbasis properti stabil dari sumber: station, normalized path, dan ukuran file. Timestamp boleh dipakai sebagai metadata, bukan discriminator utama.

### 2. UI realtime butuh invalidation yang spesifik

Beberapa review menemukan station cards/session detail bisa stale setelah `photo.routed`, `photo.quarantined`, atau `quarantine.assigned`. Query list dan detail tidak otomatis refresh bila invalidation terlalu umum atau tidak mengenai key yang benar.

**Lesson:** Setiap event SSE harus dipetakan ke query invalidation yang eksplisit: sessions list, session detail, station quarantine summary, quarantine list, dan activity log.

### 3. Persistence bukan hanya load/save; harus transactional dan actionable

Story 4.5 awalnya menambahkan persistence, lalu review menemukan runtime mutations belum selalu persisted dan partial save antar store bisa meninggalkan durable state tidak konsisten.

**Lesson:** Untuk state yang saling terkait, success API harus berarti state sudah durable. Jika persistence gagal, API perlu mengembalikan error actionable dan rollback state runtime/durable secara aman.

### 4. Locked-session recovery perlu aturan yang ketat

Manual assignment ke locked session sempat terlalu longgar untuk `no_active_session`. Review memperjelas bahwa locked-session assignment hanya aman untuk related `late_photo` recovery.

**Lesson:** “Manual recovery” tetap perlu guardrail backend. Confirmation UI tidak menggantikan eligibility rule server-side.

### 5. Traceability harus dipikirkan lintas epic

Epic 4 belum memindahkan/menyalin file; Epic 5 akan menyimpan original dan graded output. Namun traceability source→session/quarantine sudah harus lengkap sejak ingestion.

**Lesson:** Field traceability harus stabil sejak awal karena akan menjadi dasar original-first save, processing queue, retry, dan upload.

## Review Feedback Patterns

Tema feedback yang berulang:

- **Contract completeness:** OpenAPI dan runtime validators perlu selalu ikut berubah saat response shape berubah.
- **Safe identifiers:** Raw path tidak boleh menjadi SSE entity id atau public durable id.

## Risks Dibawa ke Epic 5

1. **Source JPG belum disalin ke output original**
   - Epic 4 hanya metadata ingestion/routing/quarantine.
   - Epic 5 harus memastikan original-first copy benar-benar durable sebelum grading.

2. **Processing state belum ada**
   - Routed photo sudah ada, tetapi processing queue/status/retry belum diimplementasikan.
   - Epic 5 perlu state machine processing yang tidak mengacaukan status routed/quarantine existing.

3. **Filename collision dan output path sanitization menjadi risiko utama berikutnya**
   - Epic 5 akan mulai membuat file output nyata.
   - Sanitization customer/order/station dan collision-safe naming harus menjadi guardrail awal.

4. **Persistence makin kompleks saat processing ditambahkan**
   - Photo/quarantine persistence sudah transactional.
   - Processing job persistence harus mengikuti standar yang sama agar tidak ada metadata pointing ke missing local files.

5. **LUT processor decision masih perlu dipastikan**
   - Epic 5 Story 5.2 membutuhkan library/tool LUT yang eksplisit dan terdokumentasi.

6. **Performance simulasi besar belum menjadi bukti end-to-end processing**
   - Epic 4 menguatkan ingestion/routing/recovery.
   - Target simulasi multi-station dengan banyak foto baru benar-benar lengkap setelah Epic 5 processing berjalan.
